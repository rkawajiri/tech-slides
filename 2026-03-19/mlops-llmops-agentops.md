# MLOps・LLMOps・AgentOps——AIシステム運用の進化とモニタリング設計

## はじめに

「MLOps、LLMOps、AgentOps」という言葉を聞いたことがあっても、それぞれの違いや使い分けが曖昧なまま、という方は多いのではないでしょうか。

この3つは単なるバズワードの変遷ではなく、AIシステムの「何がブラックボックスになるか」「何が失敗の原因になるか」が根本的に変わるたびに生まれた、実践的な概念の進化です。

本記事では、3つのOpsの違いと進化の流れを整理した上で、実際のシステムに組み込む際のモニタリング軸とツール選定まで解説します。

---

## 1. 3つのOpsはカテゴリではなくスペクトラム

まず結論から言うと、MLOps・LLMOps・AgentOpsは明確な境界線で区切られるカテゴリではなく、**連続したスペクトラム**として捉えるのが実態に近いです。

```
決定論的 ←─────────────────────────────→ 自律的
    │                                        │
  MLOps         LLMOps           AgentOps
    │               │                │
分類・回帰      プロンプト→応答    ツール呼び出し
  構造化出力      対話・生成        次の行動を自己決定
```

LLMを構造化出力の抽出に使えばMLOps的、対話に使えばLLMOps的、LLM自身が次のアクションを決定し始めればAgentOps的——同じLLMを使っていても、用途によって運用上の関心事は変わります。

---

## 2. 各Opsが解こうとした問題

### MLOps（〜2020年代前半）：「ノートブックが本番で動かない」

データサイエンティストの実験環境と本番環境の断絶を埋めることが出発点でした。

**前提とした世界観**

- モデルは決定論的（同じ入力 → 同じ出力）
- 評価指標は数値で明確（精度、F1、AUCなど）
- バグは再現可能でデバッグしやすい
- 出力は構造化データ（分類ラベル、数値）

**コアな関心事**

| 領域 | 内容 |
|------|------|
| データ管理 | Feature Store、データバージョニング |
| 実験管理 | ハイパーパラメータ追跡 |
| CI/CD | 再学習パイプライン、自動デプロイ |
| モニタリング | データドリフト、モデルドリフト検知 |
| サービング | バッチ vs オンライン推論 |

**代表ツール**：MLflow / Kubeflow / Airflow / Great Expectations

---

### LLMOps（2023年〜）：「プロンプトがコードになった」

GPT-4やClaudeの登場で「モデルをゼロから学習する」から「巨大な基盤モデルを活用する」にパラダイムシフト。MLOpsの概念の多くがそのまま使えなくなりました。

**MLOpsとの根本的な違い**

```
MLOps:   入力 → [自分が学習したモデル] → 出力
                  └ 重みを完全に管理

LLMOps:  入力 + プロンプト → [他者が作ったLLM] → 出力
                               └ 内部はブラックボックス
```

**LLMOps特有の関心事**

- **プロンプト管理**：バージョン管理・A/Bテスト・インジェクション対策
- **評価の難しさ**：出力が自然言語なので正解が一意に定まらない → LLM-as-a-Judge
- **コスト管理**：トークン単価 × 呼び出し回数が直接コストに
- **RAGパイプライン**：ベクトルDB管理、検索品質のモニタリング

**代表ツール**：Langfuse / LangSmith / Helicone / Ragas / Arize Phoenix

---

### AgentOps（2024年〜）：「LLMが次のアクションを自分で決める」

LLMが「単発の推論」から「自律的な行動の連鎖」をするようになったことで、また新たな課題が生まれました。

```
LLMOps:   ユーザー → LLM → レスポンス（1往復）

AgentOps: ユーザー → Agent → ツール呼び出し → 中間判断
                         ↑__________________________|
                         （何ターンになるか事前に不明）
```

**AgentOps特有の難しさ**

- **非決定的な実行フロー**：同じ入力でも実行ステップ数・経路が毎回異なりうる
- **長いトレースの可観測性**：数十〜数百ステップを追う分散トレーシング的な視点が必要
- **ループ・暴走の検知**：無限ループや大量API呼び出しへの対策
- **コストの予測不能性**：完了までのトークン数・ステップ数が事前に読めない

**分類の実用的な判断軸**

> **「LLM自身が次のアクションを決定しているか」**

これが最もシンプルな判断基準です。LLMが返答内容を決めるだけならLLMOps、次に何をするかまで自律的に決め始めたらAgentOpsに足が入ります。

## 3. 3つのOps比較まとめ

| | MLOps | LLMOps | AgentOps |
|---|---|---|---|
| **中心的な成果物** | 学習済みモデル | プロンプト＋LLM | エージェントの行動軌跡 |
| **評価の単位** | モデル精度 | レスポンス品質 | タスク達成度 |
| **実行の性質** | 決定論的 | 確率的（1ステップ） | 確率的（多ステップ） |
| **コストの予測性** | 高 | 中 | 低 |
| **デバッグの難易度** | 低 | 中 | 高 |
| **主な失敗モード** | データドリフト | Hallucination | ループ・誤ツール選択 |
| **モニタリング対象** | メトリクス | LLMスパン | エージェントトレース |

## 4. モニタリングの軸とツール

カテゴリを超えて、実際のシステムで観測すべき軸を整理します。

### 軸A：品質モニタリング

出力が「正しいか」を観測する、最も重要かつ難しい軸です。

**評価手法の選択**

```
決定論的出力（分類・抽出）
  └→ 正解ラベルとの比較で十分（MLOps的）

自然言語出力（対話・要約）
  └→ LLM-as-a-Judge
  └→ 埋め込み類似度（BERTScoreなど）
  └→ 人間評価

エージェントの行動列
  └→ トレース単位での評価
  └→ 最終成果物の品質で遡及評価
```

**主な指標**

| 指標 | 説明 | 適用 |
|------|------|------|
| Correctness | 事実として正しいか | LLM/Agent |
| Faithfulness | RAGソースに忠実か | RAG |
| Hallucination Rate | 幻覚の発生率 | LLM全般 |
| Task Completion Rate | タスクを完遂できたか | Agent |
| Step Efficiency | 目標達成に要したステップ数 | Agent |
| Tool Selection Accuracy | 適切なツールを選べたか | Agent |

---

### 軸B：コスト・効率モニタリング

**LLMコスト**

| 指標 | 説明 |
|------|------|
| Input/Output Token数 | モデル・エンドポイントごとに追跡 |
| コスト/リクエスト | 平均・P95・最大 |
| コスト/ユーザー | LTV計算に必要 |
| キャッシュヒット率 | セマンティックキャッシュの効果 |

**レイテンシ**

| 指標 | 説明 |
|------|------|
| TTFT（Time to First Token） | ストリーミングの体感速度 |
| E2E Latency | リクエスト全体の応答時間 |
| P50/P95/P99 | 外れ値の把握 |

---

### 軸C：信頼性・運用モニタリング

| 指標 | 説明 |
|------|------|
| Error Rate | タイムアウト・APIエラー率 |
| API Provider Uptime | 外部LLM APIへの依存リスク |
| Retry Rate | レート制限ヒット頻度 |
| Loop Detection | 無限ループへの陥入（Agent） |
| Max Step超過率 | 上限ステップで強制終了した割合 |

---

### 軸D：安全性モニタリング

| 指標 | 説明 |
|------|------|
| Prompt Injection検知率 | 悪意ある入力の検知 |
| Toxicity Rate | 有害コンテンツの発生率 |
| PII Leakage | 個人情報の出力検知 |
| Refusal Rate | ガードレール発動頻度（高すぎても問題） |

---

### 軸E：データ品質モニタリング

MLOps色が強い軸ですが、RAGのインデックス管理やEmbeddingドリフトなどLLM文脈でも重要になります。

| 指標 | 説明 |
|------|------|
| Feature/Label Drift | 入力・ターゲットの分布変化 |
| Embedding Drift | ベクトル空間上の入力分布変化 |
| Data Freshness | RAGインデックスの鮮度 |

---

### 軸F：ユーザー体験モニタリング

| 指標 | 説明 |
|------|------|
| Thumbs Up/Down率 | 明示的フィードバック |
| Session Length | エンゲージメントの代理指標 |
| Retry Rate | 同じ質問を言い換えた頻度（不満の代理） |
| Abandonment Rate | 途中離脱率 |

## 5. ツールマップ

### LLM/Agentトレース観測

| ツール | 特徴 | 向いている用途 |
|------|------|------|
| **Langfuse** | OSS・セルフホスト可、コスパ良好 | LLM全般、RAG |
| **LangSmith** | LangChain公式、Agent対応が厚い | LangChainユーザー |
| **Arize Phoenix** | OSS、評価・可視化が強い | 評価重視の用途 |
| **Helicone** | 軽量プロキシ型、導入が容易 | コスト監視の入口 |
| **Braintrust** | 評価パイプライン構築が強い | 評価ループを回したい |

### ML特化（MLOps寄り）

| ツール | 特徴 |
|------|------|
| **MLflow** | 実験管理・モデルレジストリの標準 |
| **Weights & Biases** | 学習曲線・ハイパーパラメータ追跡 |
| **Evidently AI** | データ・モデルドリフト検知 |
| **Great Expectations** | データ品質バリデーション |

### 安全性・ガードレール

| ツール | 特徴 |
|------|------|
| **Guardrails AI** | 入出力バリデーションフレームワーク |
| **LlamaGuard** | Meta製、有害コンテンツ分類 |
| **Presidio** | Microsoft製、PII検知 |

### インフラ・汎用可観測性

| ツール | 特徴 |
|------|------|
| **OpenTelemetry** | 業界標準トレーシング（LLM拡張仕様が策定中） |
| **Datadog** | LLM Observability機能を追加済 |
| **Grafana + Prometheus** | 自前構築の王道 |

```
コスト  GPU使用率        トークンコスト       ステップ数×コスト

ツール  MLflow/W&B       Langfuse/Helicone  LangSmith/Phoenix
```

## おわりに

MLOps・LLMOps・AgentOpsは、それぞれ独立したフレームワークではなく、AIシステムの複雑化に応じて積み上がってきた概念のスペクトラムです。

実践的には「自分のシステムがスペクトラムのどこにあるか」を軸ごとに評価し、必要な観測だけをピンポイントで設計するアプローチが合理的です。全部の軸を最初から厚くする必要はなく、**重心がどこにあるかに応じて優先度を決める**ことがポイントになります。
